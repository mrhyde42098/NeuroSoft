## 🔴 Falsos positivos investigados (descartados con razón)

### FP-1 · `const X = ExtX;` en App.jsx líneas 50-57
- **Patrón:** 5 matches `^const\s+[A-Z]\w+\s*=\s*Ext[A-Z]\w*` que el skill §audit-meta-2026-05 marca como crítico.
- **Investigación:** los 5 identificadores (`AuthProvider`, `DarkProvider`, `ToastProvider`, `A11yProvider`, `ErrorBoundary`) son re-exports intencionales con prefijo `Ext*` para marcar "vienen de un import externo en este archivo".
- **Imports verificados (línea 12-13):** `{AuthProvider as ExtAuthProvider, ...} from "./contexts.jsx"` y `import ExtErrorBoundary from "./ui/ErrorBoundary.jsx"`.
- **Comentario en código (líneas 47-49):** "Mantenemos los identificadores con re-asignaciones para no tocar las 19 páginas que aún viven en este archivo."
- **Conclusión:** Falso positivo — patrón defensivo documentado. ✓

### FP-2 · `catch {}` silenciosos (17 matches)
- **Categorías:**
  - **7** son de `localStorage.setItem` / `removeItem` en EvalApplyPage, ClinicalHistoryPage, EstudiarPage, QuizPage — patrón defensivo estándar cuando el storage puede estar en modo privado o cuota llena.
  - **5** son de `Notification` API en App.jsx, AgendaPage, ErrorBoundary — la API puede no estar disponible o tener permisos revocados.
  - **2** son de `fetch` con `.catch(() => null)` en ClinicalHistoryPage:179 — fallo de endpoint devuelve null, manejado en la misma línea.
  - **2** son de ErrorBoundary catch interno — el ErrorBoundary mismo no puede lanzar errores.
  - **1** es de safeLS:39 — ya loggea via console.debug.
- **Conclusión:** Todos los `catch {}` son **defensivos apropiados** para operaciones best-effort. No tragan errores clínicos. ✓

### FP-3 · `window.confirm` en contexts.jsx:290
- **Contexto:** Fallback del hook `useConfirm` cuando se invoca sin provider (típicamente en tests sueltos sin wrap de providers).
- **Código:** `return ctx || ((opts) => Promise.resolve(window.confirm(opts?.message || "¿Continuar?")));`
- **Comentario (líneas 282-286):** "Si por algún motivo no hay provider (tests sueltos), cae a window.confirm para no romper."
- **Conclusión:** Fallback defensivo documentado. En producción, `useConfirm` SIEMPRE devuelve el modal personalizado (`<ConfirmProvider>` está en `App.jsx` raíz). ✓

### FP-4 · 43 `except Exception:` en backend
- **Análisis:** Patrón estándar en Python para manejar errores en endpoints HTTP / I/O externo / parsing JSON.
- **Verificación:** todos los handlers en `main.py:471`, `reports.py`, `evaluations.py`, `auth.py`, etc. tienen logging adyacente o retornan respuesta HTTP apropiada (no son "tragan y siguen" peligrosos).
- **Conclusión:** Patrón defensivo apropiado para código clínico-multi-tenant. ✓
