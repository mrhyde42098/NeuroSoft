## ALTOS

### A1 · Logo/firma del PDF se pierde silenciosamente si el base64 es inválido
- **Archivo:** `neurosoft-backend/app/infrastructure/report_service.py:275` y `:899`
- **Categoría:** Error tragado sin logging
- **Código actual:**
  ```python
  except Exception:
      pass  # logo_drawn queda False, firma simplemente no se dibuja
  ```
- **Problema:** Si el profesional sube una firma/logo con base64 corrupto o formato no soportado, el PDF se genera SIN la imagen y sin ningún aviso. El clínico puede no notar que falta su firma en un informe legal.
- **Impacto clínico:** Un informe sin firma visible puede ser rechazado por una EPS o junta médica.
- **Sugerencia:** Agregar `logger.warning("Logo/firma no se pudo decodificar: %s", e)` y opcionalmente un texto placeholder ("[Firma no disponible]").

### A2 · CIE-10 lookup falla silenciosamente
- **Archivo:** `neurosoft-backend/app/infrastructure/report_service.py:950`
- **Categoría:** Error tragado sin logging
- **Código actual:**
  ```python
  except Exception:
      pass
  return ""
  ```
- **Problema:** Si el catálogo CIE-10 tiene un error de datos o la función recibe un código malformado, retorna string vacío sin loggear. Un código CIE-10 válido podría desaparecer del informe sin rastro.
- **Sugerencia:** `logger.debug("CIE-10 lookup falló para código '%s': %s", code, e)`

### A3 · `dangerouslySetInnerHTML` con regex-based markdown en ArtículosPage
- **Archivo:** `neurosoft-frontend/src/app/aprender/ArticulosPage.jsx:39` y `:57`
- **Categoría:** XSS potencial (bajo riesgo actual)
- **Código actual:**
  ```jsx
  dangerouslySetInnerHTML={{
    __html: ln.replace(/^[-*] /, "")
      .replace(/\*\*(.+?)\*\*/g, "<strong>$1</strong>")
      .replace(/`(.+?)`/g, '<code class="ns-mono ...">$1</code>'),
  }}
  ```
- **Problema:** El contenido viene de `aprenderContent.js` (datos estáticos internos, NO input de usuario), por lo que el riesgo real es mínimo. Sin embargo, si un futuro editor agrega HTML malicioso en el markdown, se renderizaría directamente.
- **Sugerencia:** Escapar `<` y `>` antes de aplicar las transformaciones regex, o usar una librería markdown segura.

## MEDIOS

### M1 · `window.confirm()` nativo en lugar de `useConfirm()` hook
- **Archivos:**
  - `neurosoft-frontend/src/app/aprender/EstudiarPage.jsx:82`
  - `neurosoft-frontend/src/app/patients/CompanionsSection.jsx:96`
- **Categoría:** Inconsistencia UX
- **Código actual:**
  ```js
  if (!window.confirm("¿Reiniciar progreso de todas las tarjetas?")) return;
  if (!window.confirm("¿Eliminar este acompañante?")) return;
  ```
- **Problema:** El proyecto tiene un `ConfirmProvider` con modal estilizado y soporte para `dangerous: true`. Estos 2 sitios usan el `window.confirm()` nativo del navegador, rompiendo la consistencia visual y la accesibilidad.
- **Sugerencia:** Reemplazar por `await confirm({ title, message, dangerous: true })`.

### M2 · 73 warnings `no-unused-vars` en ESLint
- **Archivos principales:**
  - `EvalResultsPage.jsx` (16 warnings)
  - `App.jsx` (11 warnings)
  - `ClinicalInterpretationPanel.jsx` (6 warnings)
  - `RehabPage.jsx` (4 warnings)
  - `EvalApplyPage.jsx` (3 warnings)
  - `ReactivePanel.jsx` (3 warnings)
- **Categoría:** Dead code / imports sin usar
- **Problema:** Variables e imports no usados incrementan el bundle size y confunden al lector. Algunos pueden ser aliases rotos que apuntan a nada.
- **Sugerencia:** Pasar `npx eslint src --fix` para eliminar imports no usados, y revisar manualmente las variables no usadas.

### M3 · 19 warnings `react-hooks/exhaustive-deps`
- **Archivos principales:**
  - `EvalResultsPage.jsx` (2)
  - `EvalApplyPage.jsx` (1)
  - `RehabPage.jsx` (1)
  - `PanelCompartir.jsx` (1)
  - `AdminKpisTab.jsx` (1)
  - `HistorialPage.jsx` (1)
- **Categoría:** Posibles stale closures
- **Problema:** `useEffect` con dependencias incompletas puede causar que un callback use valores obsoletos de state/props. En la mayoría de los casos están suprimidos intencionalmente (`// eslint-disable-next-line`), pero algunos podrían causar bugs sutiles de stale state.
- **Sugerencia:** Revisar cada caso individualmente. Los que tienen `// eslint-disable-next-line` son intencionales; los que no, deberían evaluarse.

### M4 · 209 colores hex hardcoded en inline styles
- **Archivos principales:** `EvalApplyPage.jsx`, `EvalResultsPage.jsx`, `InformesPage.jsx`, `ClinicalHistoryPage.jsx`, `OrdenClinicoBanner.jsx`, `CSSRSForm.jsx`, `PanelIA.jsx`
- **Categoría:** Dark mode incompleto
- **Problema:** Muchos colores son semánticos/clínicos (rojo=déficit, verde=normal, amber=alerta) y están bien como están. Pero algunos son colores de superficie (`#f5f3ef`, `#fff7ed`, `#fff8f0`, `#f0fdfa`) que no se invierten en dark mode, creando parches claros sobre fondo oscuro.
- **Sugerencia:** Los colores semánticos clínicos pueden quedarse. Los de superficie deberían migrarse a `var(--ns-subtle)`, `var(--ns-card)`, etc. Usar skill `/dark-mode-fix`.

### M5 · 97 clases Tailwind con colores hardcoded (`bg-white`, `bg-gray-50`, `text-gray-700`)
- **Archivos principales:** `EvalApplyPage.jsx`, `PanelCompartir.jsx`, `ConfigPage.jsx`, `ClinicalHistoryPage.jsx`, `PatientsPage.jsx`, `Sidebar.jsx`
- **Categoría:** Dark mode incompleto
- **Problema:** Clases como `bg-white`, `bg-gray-50`, `text-gray-700` no respetan las variables CSS de dark mode. En modo oscuro, estos elementos aparecen como parches blancos/gris claro sobre fondo oscuro.
- **Sugerencia:** Reemplazar por clases que usen CSS vars o por las variables `var(--ns-card)`, `var(--ns-subtle)`, `var(--ns-text)`.

## BAJOS

### B1 · `key={i}` con index en listas (120 instancias)
- **Archivos:** Distribuidos en ~30 archivos
- **Categoría:** React anti-pattern (bajo impacto)
- **Problema:** Usar el índice como key puede causar re-renders incorrectos si la lista se reordena. Sin embargo, la mayoría de las instancias son listas estáticas (errores, bullets, opciones de menú) que nunca se reordenan.
- **Sugerencia:** Priorizar los casos donde la lista es dinámica (agregar/eliminar/reordenar). Los estáticos pueden dejarse.

### B2 · 31 `catch {}` / `.catch(() => {})` silenciosos
- **Archivos:** Distribuidos en ~20 archivos
- **Categoría:** Error swallowing
- **Problema:** La mayoría son intencionales: `localStorage.removeItem()`, `Notification()`, `safeLS` operations. Algunos `.catch(() => {})` en API calls (DashboardPage, HistorialPage, ScreeningPage) tragan errores de red sin feedback al usuario.
- **Sugerencia:** Los de localStorage/Notification están bien. Los de API calls podrían mostrar un toast sutil o al menos `console.warn`.

### B3 · `bg-white` en toggle switches de ConfigPage
- **Archivo:** `neurosoft-frontend/src/app/config/ConfigPage.jsx:69` y `:298`
- **Categoría:** Dark mode cosmetic
- **Código actual:**
  ```jsx
  <div className={`absolute ... w-5 h-5 bg-white rounded-full shadow ...`}/>
  ```
- **Problema:** El círculo del toggle switch es siempre blanco, lo cual se ve bien en light mode pero contrasta raro en dark mode.
- **Sugerencia:** Usar `bg-white dark:bg-gray-200` o similar.

### B4 · Funciones muy largas (>150 líneas)
- **Archivos:** `EvalApplyPage.jsx` (607 líneas), `EvalResultsPage.jsx` (468 líneas), `InformesPage.jsx` (726 líneas), `PanelIA.jsx` (1179 líneas), `RehabPage.jsx` (1090 líneas), `TherapyPage.jsx` (584 líneas)
- **Categoría:** Mantenibilidad
- **Problema:** Archivos muy grandes son difíciles de auditar y mantener. Sin embargo, el patrón de minificación JSX hace que las líneas sean engañosas (una línea puede contener 500+ caracteres de JSX).
- **Sugerencia:** Extraer sub-componentes a archivos separados cuando sea práctico. No urgente.

### B5 · `except Exception: pass` en auth middleware fallback
- **Archivo:** `neurosoft-backend/app/main.py:462`
- **Categoría:** Fail-open pattern
- **Código actual:**
  ```python
  except Exception:
      return await call_next(request)
  ```
- **Problema:** Si el middleware de auth falla por una razón inesperada, la request pasa sin autenticación. Es un diseño fail-open deliberado para no bloquear la app, pero en producción podría ser un vector de bypass.
- **Sugerencia:** En producción, considerar fail-close (retornar 500) o al menos loggear el evento como warning.

### B6 · `verify_password` retorna False en cualquier excepción
- **Archivo:** `neurosoft-backend/app/infrastructure/auth/auth_service.py:105`
- **Categoría:** Fail-safe pattern (aceptable)
- **Código actual:**
  ```python
  def verify_password(plain: str, hashed: str) -> bool:
      try:
          return _get_pwd_context().verify(plain, hashed)
      except Exception:
          return False
  ```
- **Problema:** Si bcrypt falla (hash malformado, librería corrupta), el login simplemente falla sin loggear la razón. Esto es fail-safe correcto (no permite acceso), pero dificulta debugging.
- **Sugerencia:** Agregar `logger.debug("verify_password falló: %s", e)` antes del return.
