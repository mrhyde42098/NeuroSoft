## 🔴 CRÍTICOS (1)

### C1 · `EdadSexoStrategy` retorna falso positivo 0.0 cuando sigma=0
- **Archivo:** `neurosoft-backend/app/domain/clinical_engine/strategies.py:490`
- **Categoría:** División por cero clínica / falso positivo
- **Código actual:**
  ```python
  z = round((float(pd) - mu) / sigma, 2) if sigma != 0 else 0.0
  ```
- **Problema:** Misma inconsistencia que `ZScoreMultipleStrategy` tenía antes del fix del 31 mayo. Cuando `sigma=0` (baremo corrupto), retorna `z=0.0` = "rendimiento promedio" en lugar de marcar `sin_norma`. La strategy hermana `ZScoreStrategy` (línea 291) YA usa el patrón correcto: `puntaje_escalar=None, metadata={"sin_norma": True}`.
- **Impacto clínico:** CDI (Depresión Infantil) — única prueba que usa `EdadSexoStrategy`. Si el baremo está corrupto, un niño con depresión severa aparecería como "Promedio" en lugar de tener una advertencia.
- **Sugerencia:** Replicar el patrón §H9-fix de `ZScoreStrategy:291-298`: early return con `puntaje_escalar=None` y `sin_norma: True`.

## 🟠 ALTOS (5)

### A1 · `verify_password()` traga todas las excepciones sin log
- **Archivo:** `neurosoft-backend/app/infrastructure/auth/auth_service.py:104`
- **Categoría:** Logging silencioso en seguridad
- **Código actual:**
  ```python
  except Exception:
      return False
  ```
- **Problema:** Si bcrypt falla por cualquier razón (memoria, archivo corrupto), el sistema dice "contraseña incorrecta" sin registrar el error real. El clínico no puede acceder y no hay forma de diagnosticar por qué.
- **Impacto:** Un clínico legítimo bloqueado sin explicación. El desarrollador (Johan) no sabe si es bug o ataque.
- **Sugerencia:** Agregar `logger.warning("verify_password falló: %s", exc)` antes del `return False`.

### A2 · 5 `api.get().catch(() => {})` fallan sin feedback al usuario
- **Archivos:**
  - `DashboardPage.jsx:74` — plan de rehabilitación
  - `EvalApplyPage.jsx:52` — carga de pacientes
  - `ClinicalHistoryPage.jsx:167` — carga de panel de pacientes
  - `ConfigPage.jsx:54` — eliminar profesional
  - `ClinicalHistoryPage.jsx:188` — búsqueda CIE-10
- **Categoría:** UX/silent failure
- **Problema:** Si el backend está caído o el JWT expiró, la UI se queda vacía sin explicación. El clínico no sabe si es problema suyo, del sistema, o simplemente no hay datos.
- **Impacto:** Clínico cree que la app "no funciona" cuando en realidad es un error de red/permissions. Aumenta llamadas de soporte.
- **Sugerencia:** Cambiar a `.catch(() => toast.error("Error al cargar [recurso]"))`. Algunos ya se arreglaron en el sprint anterior, estos 5 quedaron.

## 🟡 MEDIOS (6)

### M1 · `EscolaridadMapper` asume "Secundaria" sin advertir
- **Archivo:** `neurosoft-backend/app/core/utils.py:414, 423`
- **Categoría:** Validación de datos clínicos
- **Código actual:**
  ```python
  return _EDU_CODE_MAP.get(nivel, "S")  # fallback silencioso
  ```
- **Problema:** Si un clínico escribe mal el nivel educativo (ej. "Universitaro" en vez de "Universitario"), el sistema asume Secundaria sin decir nada. Los baremos de Adulto Mayor dependen críticamente de la escolaridad.
- **Impacto clínico:** Un paciente universitario podría recibir baremos de secundaria, alterando su perfil neuropsicológico.
- **Sugerencia:** Loguear `logger.warning("Escolaridad no reconocida: '%s', usando fallback 'S'", nivel)` o lanzar advertencia al frontend.

### M2 · Inputs de PD sin validación de rango en frontend
- **Archivos:** `EvalApplyPage.jsx:556-561` (tabla de registro), `ReactivePanel.jsx` (items)
- **Categoría:** UX/validación
- **Problema:** Los inputs `<input type="number">` no tienen `min` ni `max`. Un clínico puede ingresar `-5`, `99999`, o texto. El backend valida y rechaza, pero el error llega tarde (al guardar/calificar).
- **Impacto:** UX frustrante si el clínico descubre el error al final de la evaluación.
- **Sugerencia:** Agregar `min={0}` a todos los inputs de PD. Para pruebas con rango máximo conocido (ej. WISC-IV Claves max=135), agregar `max={valor}`.

### M3 · Sin validación de que el clínico haya completado pasos obligatorios antes de generar informe
- **Archivos:** `EvalResultsPage.jsx` (flujo de generación), `reports.py:425` (completitud)
- **Categoría:** Flujo clínico
- **Problema:** Aunque `InformesPage.jsx` muestra indicadores de completitud, el botón "Generar PDF" en `EvalResultsPage` no bloquea si faltan secciones críticas (firma, impresión diagnóstica). Solo `InformesPage` tiene ese bloqueo.
- **Impacto:** El clínico puede generar un PDF incompleto desde la pantalla de resultados sin darse cuenta.
- **Sugerencia:** Replicar la validación de completitud de `InformesPage` en `EvalResultsPage`.

### M4 · Strings "N/A" como sentinel de campo vacío
- **Archivo:** `ClinicalHistoryPage.jsx:155-157`
- **Código actual:**
  ```js
  [f.patologicos_medicos, f.farmacologicos, f.familiares].filter(v=>v&&v!=="N/A"&&v!=="").length
  ```
- **Problema:** El sistema usa `"N/A"` como marcador de "sin datos". Esto es frágil: si un clínico escribe "N/A" como parte de una nota real (ej. "El paciente reporta N/A en pruebas previas"), el sistema lo cuenta como vacío.
- **Impacto:** Bajo — escenario improbable, pero el patrón es propenso a errores.
- **Sugerencia:** Usar `null` como sentinel en lugar de string mágico, o un campo booleano `is_empty`.

### M5 · Contraseñas: longitud mínima 4 para login
- **Archivo:** `neurosoft-backend/app/presentation/api/v1/auth.py:42`
- **Categoría:** Seguridad
- **Problema:** `min_length=4` para el endpoint de login es bajo. Aunque es solo login (no registro), permite contraseñas triviales como "1234".
- **Impacto:** En una app desktop offline es menor, pero si se expone a red local, es vulnerable.
- **Sugerencia:** Aumentar a `min_length=6` para consistencia con registro.

### M6 · `ComparativoStrategy` (línea 602-637) — type hints inconsistentes
- **Archivo:** `neurosoft-backend/app/domain/clinical_engine/strategies.py:602-637`
- **Categoría:** Código
- **Código actual:**
  ```python
  sp, sc, spc = float(valor[0]), float(valor[1]), float(valor[2])
  ```
- **Problema:** Asume que `valor` tiene exactamente 3 elementos. Si el baremo tiene 2 o 4, crashea con `ValueError` (no manejado). No tiene validación de `len(valor) >= 3`.
- **Impacto:** Si un baremo futuro tiene formato diferente, crashea al calificar.
- **Sugerencia:** Validar `if len(valor) < 3: return _not_found(...)`.

## 🎯 Mejoras recomendadas (perspectiva clínica)

### 1. Informes: agregar percentiles a la tabla de resultados
La tabla de resultados en el PDF muestra Z y clasificación, pero no percentiles. Los percentiles son más intuitivos para padres y pacientes ("su hijo está en el percentil 75"). Implementación: agregar columna `percentil` a `ResultadoPruebaDTO` calculado desde Z.

### 2. Screening: agregar BDI-II y MOCA-BASIC
El catálogo de screening (`SCREENING_FORMS`) cubre MMSE, MoCA, PHQ-9, GAD-7, BAI, HADS, Barthel, Lawton. Falta:
- **BDI-II** (Beck Depression Inventory) — depression en adultos
- **MOCA-BASIC** — tamizaje cognitivo para baja escolaridad

### 3. Orden clínico: integrar los dos sistemas
Actualmente hay DOS sistemas de sugerencia de orden:
- `getSuggestedClinicalTest()` en `clinical.js` (sub-header con botón "Ir a esta")
- `PROTOCOLOS_ORDEN` en `protocolosOrden.js` (banner con timer/interferencia)

Deberían unificarse en un solo sistema que cubra ambos casos.

### 4. Dashboard: agregar "Pacientes sin informe" KPI
El dashboard muestra pacientes totales, evaluaciones, tendencias. Sería útil un KPI de "evaluaciones sin informe generado" para que el clínico sepa qué tiene pendiente.

### 5. Backups: verificar integridad periódicamente
Aunque el backup automático diario existe (02:00), no hay verificación periódica de que los backups antiguos sigan siendo legibles. Un backup de hace 30 días podría estar corrupto y nadie lo sabría hasta necesitarlo.
